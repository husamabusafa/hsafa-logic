# Per-Resource API Keys Architecture

> How per-resource API keys secure the Hsafa platform in a multi-tenant production environment.

---

## The Problem

A single global `CORE_API_KEY` is fine during development. In production with multiple users, we need:

- Users can't access each other's haseefs or scopes
- External scope services authenticate without seeing the platform's master key
- Attaching a scope to a haseef requires proof of ownership of **both**

---

## The Model

Every haseef and every scope gets its own API key, generated by Core.

```
┌─────────────────────────────────────────────────────┐
│                      Hsafa Core                      │
│                                                      │
│  Haseef "Ali"    →  hsk_haseef_abc123               │
│  Haseef "Sara"   →  hsk_haseef_def456               │
│                                                      │
│  Scope "spaces"      →  hsk_scope_sp_001            │
│  Scope "scheduler"   →  hsk_scope_sc_002            │
│  Scope "my-weather"  →  hsk_scope_mw_003            │
│                                                      │
│  Attach(Ali ↔ my-weather) = needs BOTH keys         │
└─────────────────────────────────────────────────────┘
```

---

## Key Types

| Key Type | Format | Who Holds It | What It Can Do |
|----------|--------|-------------|----------------|
| **Haseef Key** | `hsk_haseef_*` | Haseef owner (stored in Spaces DB) | Manage haseef, attach/detach scopes, push sense events |
| **Scope Key** | `hsk_scope_*` | Scope owner (stored in Spaces DB or user's .env) | Connect SSE, register tools, post tool results — for this scope only |
| **Service Key** | `hsk_service_*` | Spaces server (never exposed to users) | Backend-to-Core admin operations (built-in scopes, instruction sync) |

---

## Key Lifecycle

### Haseef Key — Created When User Creates a Haseef

```
User creates haseef "Ali" via Spaces UI
  → Spaces server calls Core: POST /api/haseefs
  → Core generates key: hsk_haseef_abc123
  → Core returns: { haseef, apiKey: "hsk_haseef_abc123" }
  → Spaces stores apiKey in UserApiKey table, linked to userId
  → User never sees this key (Spaces manages it behind the scenes)
```

### Scope Key — Created When a Scope Is Registered

**Built-in scopes (spaces, scheduler, postgres):**
```
Spaces server starts up
  → For each built-in scope, uses the Service Key to register with Core
  → Core returns scope keys
  → Spaces stores them internally (users never see these)
```

**External scopes (user-deployed):**
```
User creates scope via CLI: hsafa scope create my-weather
  → CLI calls Core: POST /api/scopes
  → Core generates key: hsk_scope_mw_003
  → Core returns: { scope, apiKey: "hsk_scope_mw_003" }
  → CLI displays key to user (one-time display)
  → User saves it in their service's .env

User registers scope in Spaces UI:
  → Enters scope name + scope key
  → Spaces validates key against Core
  → Spaces stores key in UserApiKey table
  → Scope appears in "My Instances" for attach/detach
```

## Attaching = Prove You Own Both

To attach a scope to a haseef, you must present both keys:

```http
POST /api/haseefs/:haseefId/scopes/:scopeName/attach
Headers:
  x-haseef-key: hsk_haseef_abc123
  x-scope-key: hsk_scope_mw_003
```

Core validates both keys → creates the link. No one can attach a scope to someone else's haseef.

When Spaces UI does the attach, it already has both keys in its DB (stored when the user created/registered each resource). The user just clicks "Attach" and Spaces sends both behind the scenes.

### Detaching

```http
POST /api/haseefs/:haseefId/scopes/:scopeName/detach
Headers:
  x-haseef-key: hsk_haseef_abc123
```

Only the haseef key is needed to detach — you own the haseef, you decide what's attached to it.

## What Each Key Authorizes

### Haseef Key (`hsk_haseef_*`)

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/haseefs/:id` | GET | Get haseef details |
| `/api/haseefs/:id` | PUT | Update haseef config |
| `/api/haseefs/:id/scopes/:scope/attach` | POST | Attach scope (also needs scope key) |
| `/api/haseefs/:id/scopes/:scope/detach` | POST | Detach scope |
| `/api/haseefs/:id/senses` | POST | Push sense events |
| `/api/haseefs/:id/scopes` | GET | List attached scopes |

### Scope Key (`hsk_scope_*`)

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/scopes/:scope/stream` | GET | SSE stream — receive tool calls |
| `/api/haseefs/:id/scopes/:scope/tools` | PUT | Register/update tools |
| `/api/haseefs/:id/tools/:callId/result` | POST | Post tool execution result |
| `/api/scopes/:scope` | GET | Get scope info |

### Service Key (`hsk_service_*`)

| Endpoint | Method | Description |
|----------|--------|-------------|
| All endpoints | * | Full admin access (Spaces backend only) |

## Spaces Key Storage

### Prisma Model

```prisma
model UserApiKey {
  id           String   @id @default(cuid())
  userId       String
  resourceType String   // "haseef" | "scope"
  resourceId   String   // haseef ID or scope name
  apiKey       String   // the Core-issued key (encrypted at rest)
  createdAt    DateTime @default(now())

  user User @relation(fields: [userId], references: [id])

  @@unique([userId, resourceType, resourceId])
}
```

### Example Data

| userId | resourceType | resourceId | apiKey |
|--------|-------------|------------|--------|
| user_1 | haseef | haseef-ali-uuid | hsk_haseef_abc123 |
| user_1 | haseef | haseef-sara-uuid | hsk_haseef_def456 |
| user_1 | scope | scheduler | hsk_scope_sc_002 |
| user_1 | scope | my-weather | hsk_scope_mw_003 |
| user_2 | haseef | haseef-omar-uuid | hsk_haseef_ghi789 |

### How Spaces Uses Stored Keys

```typescript
// When user clicks "Attach" in the UI:
const haseefKey = await getStoredKey(userId, "haseef", haseefId);
const scopeKey  = await getStoredKey(userId, "scope", scopeName);

await fetch(`${CORE_URL}/api/haseefs/${haseefId}/scopes/${scopeName}/attach`, {
  method: "POST",
  headers: {
    "x-haseef-key": haseefKey,
    "x-scope-key": scopeKey,
  },
});

// When built-in scopes register tools (server startup):
// Uses the Service Key — no per-user key needed
await fetch(`${CORE_URL}/api/haseefs/${haseefId}/scopes/spaces/tools`, {
  method: "PUT",
  headers: { "x-api-key": SERVICE_KEY },
  body: JSON.stringify({ tools, instructions }),
});
```

## Key Rotation

Users can regenerate a key if it's compromised:

```bash
# Rotate a haseef key
hsafa key rotate --haseef <haseef-id>
# → Old key invalidated immediately
# → New key: hsk_haseef_xyz789
```

When Spaces rotates a key:
1. Calls Core: `POST /api/keys/:keyId/rotate`
2. Core invalidates old key, generates new one
3. Spaces updates its `UserApiKey` record
4. If it's a scope key for an external service, user must update their `.env`

---

## Security Considerations

- **Encryption at rest** — `UserApiKey.apiKey` is encrypted in the DB (never stored as plaintext)
- **Key isolation** — a haseef key can't access another haseef; a scope key can't access another scope
- **Rate limiting** — Core rate-limits per key to prevent abuse
- **Key expiration** — optional TTL on keys (e.g., scope keys expire after 90 days if not rotated)
- **Audit log** — Core logs which key was used for each API call
- **Scope key never in UI** — Spaces stores it but never displays it in the dashboard (only CLI shows it once on creation)
- **Service key protected** — never exposed in API responses, env vars only on the server

---

## Implementation Order

### Phase 1 — Core Changes
1. Generate per-haseef API keys when haseefs are created
2. Generate per-scope API keys when scopes are registered
3. Add key validation middleware (check `x-haseef-key`, `x-scope-key`, `x-api-key`)
4. Attach endpoint requires both keys

### Phase 2 — Spaces Changes
5. Add `UserApiKey` Prisma model + migration
6. Store haseef keys when creating haseefs
7. Store scope keys when registering external scopes
8. Update all Core API calls to use stored keys
9. Update attach/detach routes to send both keys
